# 项目完成报告（覆盖第一阶段与第二阶段）

## 1. 报告说明与项目目标

本报告按照《大作业一.docx》第一阶段“项目报告”要求组织撰写，覆盖项目目标、系统设计、测试、AI 使用、问题与解决等核心内容，并在同一框架下完整呈现第一阶段与第二阶段的实施结果。

项目目标是开发一个基于 MongoDB 的在线问卷系统，并按真实迭代场景分阶段推进：

- **第一阶段**：完成问卷基础业务闭环（账号、问卷、题型、跳转、提交、统计）。
- **第二阶段**：在第一阶段基础上按《需求变更》扩展题库能力（复用、共享、版本链、回退、使用追踪、跨问卷统计）。

课程评分重点对应为：数据库设计、后端业务逻辑、测试质量、文档质量与 AI 使用说明。

## 2. 第一阶段设计与实现

### 2.1 系统架构

- 前端：`React + Vite + TypeScript`
- 后端：`Node.js + Express + TypeScript`
- 数据库：`MongoDB + Mongoose`

后端采用 `Route -> Controller -> Service -> Model` 分层结构；前端分为管理端与填写端。系统设计原则为：**前端负责交互体验，后端负责数据真实性裁决**。

### 2.2 第一阶段核心功能

1. 用户注册与登录（JWT 鉴权）
2. 问卷创建、编辑、发布、关闭与列表管理
3. 题型支持：单选、多选、文本、数字
4. 跳转逻辑：`logicRules + defaultNextQuestionId`
5. 提交校验：题型合法性、区间校验、必答校验、匿名权限校验
6. 统计查询：单选/多选统计、文本汇总、数字平均值

## 3. 第二阶段需求变更分析

第一阶段完成后，用户反馈主要集中在“题目不可复用、协作不便、改题影响历史问卷、版本不可追溯”等问题。需求变更将其细化为以下能力：

1. 保存常用题目并复用  
2. 题目共享  
3. 改题不影响既有问卷  
4. 版本历史与回退  
5. 同题多版本并行  
6. 查看题目被哪些问卷引用  
7. 题库化管理  
8. 单题跨问卷统计

第二阶段严格遵循“在第一阶段基础上增量演进”的约束，不推倒重做。

## 4. 第二阶段设计演进

### 4.1 数据结构演进

在 `Users / Surveys / Responses` 基础上新增 `QuestionTemplates` 集合，核心字段包括：

- `rootTemplateId`：版本链标识
- `version`：版本号
- `previousTemplateId`：前置版本关联
- `versionRemark`：版本备注（更新/回退说明）
- `sharedWithUserIds`：共享对象

问卷题目新增关联元数据：

- `questionTemplateId`
- `questionTemplateVersion`
- `description`（题目补充说明）

### 4.2 后端能力扩展

新增并迭代以下接口族：

- 题库 CRUD：`/api/questions`
- 共享管理：`/api/questions/:id/shares`
- 版本历史：`/api/questions/:id/versions`
- 历史回退：`/api/questions/:id/restore`
- 使用追踪：`/api/questions/:id/usages`
- 跨问卷统计：`/api/questions/:id/statistics`

核心语义约束如下：

- 编辑题目生成新版本，不覆盖旧版本；
- 回退语义为“当前最新版本回退到历史内容，并生成新的最新版本”；
- 题库主列表仅展示每条版本链的当前版本，历史版本在历史面板查看；
- 删除题目时删除整条版本链，避免旧版本回显。

### 4.3 前端能力扩展

- 新增题库管理页（主页入口 + 编辑器选题入口）
- 支持题目创建、编辑、删除、共享
- 支持版本历史查看与回退
- 支持问卷引用关系查看
- 支持单题跨问卷统计查看
- 支持版本备注录入与展示（保留“第 N 版”标识）

## 5. 兼容性与旧功能保持

为满足“在原系统上演进”的要求，项目保留第一阶段核心结构与行为：

1. 原有问卷创建、发布、填写、统计接口持续可用；
2. 题目升级采用“新增版本”而非“原地覆盖”，已发布问卷不受影响；
3. 使用关系查询兼容历史数据：缺失 `questionTemplateId` 时按“标题 + 题型”推断；
4. 提交端仍由后端重演路径并拦截幽灵答案，原有数据防线保持不变。

## 6. AI 使用过程（摘要）

项目按课程要求使用 AI 工具辅助开发（完整日志见 `AI_USAGE_LOG.md`）。总体分工如下：

- AI 负责：初稿生成、类型与接口草案、测试样例草案、文档结构建议；
- 人工负责：架构决策、业务语义裁定、兼容策略设计、关键问题修复与最终验收。

典型人工纠偏包括：

1. 回退语义从“旧版本变新版本”修正为“当前版本回退到历史内容”；
2. 共享列表改为后端真实数据回读，修复前端状态残留；
3. 使用关系补充历史数据推断，修复引用结果缺失问题。

## 8. 开发中遇到的问题与解决

1. **运行环境不稳定**：出现过 `500` 与 `ERR_CONNECTION_REFUSED`。  
   解决：固定“MongoDB -> 后端 -> 前端”启动顺序并复核环境变量。

2. **推送与网络异常**：`git push` 曾受网络与 TLS 配置影响。  
   解决：统一 Git 网络配置并校正 SSL 后端参数，恢复稳定推送。

3. **版本语义偏差**：前端回退行为与后端版本链定义不一致。  
   解决：统一为“最新版本回退到历史内容”语义，并同步修正文案、测试与服务实现。

4. **引用追踪空结果**：历史问卷缺少模板关联字段。  
   解决：补充“标题 + 题型”兼容匹配逻辑，保证历史数据可检索。

5. **填写端状态机问题**：出现分页总数预演偏差、提交按钮锁死等问题。  
   解决：修复 `survey-engine` 路径预演与 `desk-scene` 翻页状态回收逻辑，并补充前端回归测试。
