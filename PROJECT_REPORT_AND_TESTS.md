# 项目完成报告（覆盖第一阶段与第二阶段）

## 1. 报告说明与项目目标

本报告按照《大作业一.docx》第一阶段“项目报告”要求组织（目标、设计、测试、AI 使用、问题与解决），并在同一框架下完整呈现项目第一、第二阶段的总体完成情况。

项目目标是开发一个基于 MongoDB 的在线问卷系统，并按真实迭代场景分两阶段完成：

- **阶段一**：实现问卷基础闭环（账号、问卷、题型、跳转、提交、统计）。
- **阶段二**：在阶段一基础上按《需求变更》扩展题库能力（复用、共享、版本链、回退、使用追踪等）。

课程评分重点对应为：MongoDB 设计、后端逻辑、测试、文档、AI 使用说明。

## 2. 阶段一设计与实现

### 2.1 系统架构

- 前端：`React + Vite + TypeScript`
- 后端：`Node.js + Express + TypeScript`
- 数据库：`MongoDB + Mongoose`

后端采用 `Route -> Controller -> Service -> Model` 分层，前端分为管理后台与填写端。核心原则是：**前端负责体验，后端负责真实性裁决**。

### 2.2 阶段一核心功能

1. 用户注册与登录（JWT 鉴权）
2. 问卷创建、编辑、发布、关闭、列表管理
3. 题型支持：单选、多选、文本、数字
4. 跳转逻辑：`logicRules + defaultNextQuestionId`
5. 提交校验：题型合法性、区间校验、必答校验、匿名权限校验
6. 统计查询：单选/多选统计、文本汇总、数字平均值

## 3. 阶段二需求变更分析

阶段一上线后，用户反馈集中在“题目不能复用、团队协作困难、改题影响历史问卷、版本不可追踪”四类问题。需求变更文档进一步细化为 8 项能力：

1. 保存常用题目并复用  
2. 题目共享  
3. 改题不影响既有问卷  
4. 版本历史与回退  
5. 同题多版本并行  
6. 查看被哪些问卷引用  
7. 题库化管理  
8. 单题跨问卷统计

阶段二坚持“不得推倒重做”，所有改动都在阶段一代码基础上增量实现。

## 4. 阶段二设计演进

### 4.1 数据结构演进

在 `Users / Surveys / Responses` 基础上新增 `QuestionTemplates` 集合，关键字段：

- `rootTemplateId`：版本链标识
- `version`：版本序号（第 N 版）
- `previousTemplateId`：前置版本关系
- `versionRemark`：版本备注（更新/回退说明）
- `sharedWithUserIds`：共享对象

问卷题目新增关联元数据：

- `questionTemplateId`
- `questionTemplateVersion`
- `description`（题目备注）

### 4.2 后端能力扩展

新增并迭代如下接口族：

- 题库 CRUD：`/api/questions`
- 共享：`/api/questions/:id/shares`
- 版本历史：`/api/questions/:id/versions`
- 历史回退：`/api/questions/:id/restore`
- 使用追踪：`/api/questions/:id/usages`
- 跨问卷统计：`/api/questions/:id/statistics`

关键语义：

- 编辑题目生成新版本，不覆盖旧版本；
- 回退语义为“当前最新版本回退到历史内容，并生成新的最新版本”；
- 题库主列表仅展示每条版本链的当前版本，旧版本隐式保留在历史面板；
- 删除题目删除整条版本链，避免旧版本重新暴露。

### 4.3 前端能力扩展

- 新增题库管理页（主页入口 + 编辑器选题跳转）
- 支持创建/编辑/删除/共享
- 支持版本历史查看与回退
- 支持查看问卷引用关系
- 支持查看单题跨问卷统计结果
- 支持版本备注输入与展示（并保留“第 N 版”标识）

## 5. 兼容性与旧功能保持

为满足“在原系统上演进”的约束，本项目保留了阶段一核心结构与行为：

1. 原有问卷创建、发布、填写、统计接口保持可用
2. 题目升级采用“新增版本”而非“原地覆盖”，已发布问卷不受影响
3. 使用关系查询兼容历史数据（缺失 `questionTemplateId` 时按标题+题型推断）
4. 提交端仍由后端重演路径并拦截幽灵答案，旧防线不回退

## 6. AI 使用过程（摘要）

本项目按课程要求使用 AI 工具参与开发，完整日志见 `AI_USAGE_LOG.md`。总体分工如下：

- AI 负责：初稿生成、类型与接口对齐建议、测试样例草案
- 人工负责：架构决策、语义裁定、兼容策略、关键修复与最终验收

典型人工纠偏包括：

1. 回退语义从“旧版本变新版本”纠正为“当前版本回退到历史内容”
2. 共享列表改为以后端真实数据回读，修复前端状态残留
3. 使用关系增加历史数据推断，修复引用列表空白

测试明细（步骤/输入/输出/结果）已整理在 `TEST_CASES.md`，包含阶段一与阶段二能力。

## 8. 开发中遇到的问题与解决

1. **运行环境不稳定**：出现过 `500`、`ERR_CONNECTION_REFUSED`；通过固定“Mongo -> 后端 -> 前端”重启顺序解决。
2. **网络推送失败**：`git push` 因代理端口不一致失败；统一 Git 代理配置后恢复。
3. **版本语义偏差**：恢复按钮语义与后端链路定义不一致；统一为“最新回退”语义并改文案、测试、服务实现。
4. **引用追踪空结果**：历史问卷缺模板 ID；补充标题+题型兼容匹配。
