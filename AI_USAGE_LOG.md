# AI_USAGE_LOG.md

## 1. AI 使用日志

以下记录基于本项目真实协同过程整理，重点描述“工具、提示词、产出、人工决策”四个维度。

### 场景一：问卷路径重演与防伪造架构设计

- 使用工具：
  - Gemini（统筹讨论）
  - Codex（代码设计与实现）
- 提示词概括：
  - 讨论如何让前端拥有沉浸式分页体验，同时保证跳题路径不能被伪造。
- 得到什么代码 / 方案：
  - 形成了“确定性引擎 + 演出层分离”的方案。
  - Codex 输出了 `replayFromStart`、`buildSubmitPayload`、`Reducer + Context` 的前端状态机草案。
  - 后端采用从入口题开始的全量路径重演逻辑，提交时拒绝不在真实路径上的答案。
- 自己怎么修改 / 决策：
  - 主理人拍板保留前端体验级重演，但规定后端为最终裁决者。
  - 最终实现中，前后端都保留了路径重演语义，前端负责体验，后端负责防伪造。

### 场景二：TypeScript / API 契约与联调修复

- 使用工具：
  - Codex
- 提示词概括：
  - 将前端从 Mock 模式切换为真实 `/render` 和 `/submit` 接口，并处理错误消息回显。
- 得到什么代码 / 方案：
  - 前端 `api.ts` 中实现了 Axios 客户端、统一 envelope 解包和 `ApiClientError`。
  - `vite.config.ts` 中补齐了本地 `/api` 代理，避免开发期 CORS 问题。
  - 页面从 URL `surveyId` 动态读取问卷 ID，而不是继续写死 Mock 问卷。
- 自己怎么修改 / 决策：
  - 明确要求以前后端现有 `API.md` 和 `PROJECT.md` 为契约基准，不追加未实现字段。
  - 发现浏览器端真实联调需要代理后，决定用 Vite 代理而不是修改后端业务代码。

### 场景三：活页羊皮卷视觉系统重构

- 使用工具：
  - Codex
- 提示词概括：
  - 将前端从廉价的“桌面拟物 UI”重构为克制拟物化的骨白纸张、浮雕选项和墨水质感系统。
- 得到什么代码 / 方案：
  - 形成了 `paper-sheet / paper-noise / paper-emboss / paper-deboss / parchment-smudge` 等样式系统。
  - 单选、多选、文本、数字题全部改为纸本浮雕交互。
  - 翻页动画保留 `framer-motion`，但视觉上收敛为干净的纸页翻转。
- 自己怎么修改 / 决策：
  - 主理人否决了早期“探长桌面”“铁夹子”“木纹背景”等低质量拟物方向。
  - 最终视觉参数由主理人明确指定：暖灰底、骨白纸、深海蓝墨水、极弱墨晕、克制动效。

### 场景四：集成测试与统计管道验证

- 使用工具：
  - Codex
- 提示词概括：
  - 为后端补足集成测试，验证从创建、发布、提交到统计的完整闭环。
- 得到什么代码 / 方案：
  - 生成了 `tests/integration.test.ts`。
  - 覆盖注册登录、发布、渲染、合法提交、非法数值、幽灵答案、匿名权限等路径。
  - 统计接口通过 MongoDB 聚合管道计算 `optionCounts`、`average` 和 `textValues`。
- 自己怎么修改 / 决策：
  - 结合课程要求，要求测试不仅验证成功路径，还必须覆盖非法路径与防线场景。
  - 在最终验收前又追加了一次真实浏览器端手工联调，确认答卷确实入库。

## 2. AI 使用说明总结

### 2.1 AI 帮了什么

- 快速生成了后端项目骨架与前端工程基础结构。
- 在 TypeScript 类型、请求封装、状态机拆分、DTO 对齐方面显著提速。
- 在复杂 CSS、Framer Motion 过渡、纸张浮雕交互样式上提供了高密度初稿。
- 在 MongoDB 文档模型与聚合统计管道上，帮助快速收敛为可运行实现。

### 2.2 AI 做错了什么

- 早期视觉方案曾偏向“套皮网页”或“廉价拟物化”，与目标气质不符。
- 某些前端交互初稿存在过度装饰的问题，例如木纹背景、铁夹、涂鸦式道具感过强。
- 联调脚本和自动化命令层面也曾出现过引号与环境不稳定的问题，需要人工判断和纠偏。

### 2.3 自己改了什么 / 设计了什么

- 主理人最终确立了“后端路径重演是唯一真实裁决”的架构原则。
- 明确要求采用克制拟物化，而不是廉价的视觉噪音堆叠。
- 对 UI 做了数轮方向收敛，最终锁定为“活页羊皮卷”视觉系统。
- 在前后端契约层面，坚持以现有代码实现与课程文档为准，不虚构未实现功能。

## 3. 结论

本项目中的 AI 不是独立完成者，而是高强度协作工具：

- 它最擅长快速产出结构化初稿、样板代码、样式方案和重构建议。
- 它不擅长直接替代主理人的审美决策、架构拍板与最终取舍。

最终可交付结果来自“AI 生成 + 人工审查 + 方向纠偏 + 代码落地”的组合过程。
